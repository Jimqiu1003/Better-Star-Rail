## 0. 文档结论

Better Star Rail v0.1 定位为一个**本地运行、基于屏幕视觉识别与键鼠模拟的日常任务自动化工具**，目标是帮助玩家完成重复性较高的单人日常操作，包括启动游戏、消耗开拓力、刷副本与战斗、领取委托和各类奖励，以及通过 OCR 和图像识别执行界面操作。

v0.1 不追求覆盖全部地图、全部活动和全部账号状态，而是优先完成一条**稳定、可中止、可恢复、可审计的“每日一条龙”主流程**。

项目明确接受以下事实：

1. 使用第三方自动化工具可能违反游戏服务条款，并可能导致警告、限制或封禁。
2. 项目声明、开源协议和免责条款不能消除用户账号风险，也不能替代法律意见。
3. 项目不得包含绕过反作弊、进程注入、内存读写、封包修改、隐藏进程或规避检测功能。
4. 所有自动操作必须允许用户立即中止，涉及稀有资源、星琼、燃料、付费货币或不可逆操作时必须默认禁止或二次确认。

## 1. 产品背景

《崩坏：星穹铁道》的日常流程包含启动客户端、进入游戏、消耗开拓力、重复战斗、重新派遣委托、领取每日训练、邮件和活动奖励等操作。单次耗时不一定长，但频率高、重复度高，容易出现忘记领取、开拓力溢出和重复点击疲劳。

现有参考项目证明了以下技术路线具有可行性：

- 通过屏幕截图、OCR 和模板识别判断当前界面；
- 通过键鼠模拟完成交互；
- 使用任务调度器串联多个自动化模块；
- 使用遮罩层、日志和识别结果提升可观察性；
- 通过模块化任务降低单个流程失效对全局的影响。

Better Star Rail 将借鉴参考项目的工程组织和交互思路，但不直接照搬其品牌、素材、代码或未经许可的资源。

## 2. 产品愿景

让用户用一次明确配置和一次启动，完成当天可安全自动化的重复任务；在出现异常、识别不确定或资源风险时，工具宁可停止，也不盲目点击。

### 2.1 核心价值

- **省时间**：减少每日重复操作。
- **低学习成本**：首次配置后可以一键运行。
- **透明可控**：用户能看到当前状态、识别结果和操作日志。
- **安全优先**：不执行未经确认的高风险资源操作。
- **可维护**：游戏更新后可以通过资源包和流程配置适配，而不是修改全部代码。

### 2.2 一句话定义

> Better Star Rail 是一个基于本地计算机视觉和键鼠模拟的《崩坏：星穹铁道》日常任务自动化工具。

## 3. 目标与非目标

### 3.1 v0.1 产品目标

| 编号 | 目标 | 成功标准 |
|---|---|---|
| G1 | 完成可用的每日一条龙流程 | 在支持的标准环境中，连续 20 次运行成功率不低于 85% |
| G2 | 自动消耗开拓力 | 可按用户配置选择副本、次数或资源阈值 |
| G3 | 自动完成重复战斗 | 能识别进入战斗、战斗中、结算、失败和体力不足状态 |
| G4 | 自动领取常用奖励 | 委托、每日训练、邮件、活动红点可独立启用 |
| G5 | 建立稳定的视觉自动化底座 | OCR、模板匹配、截图、点击、等待、重试可复用 |
| G6 | 保证随时可控 | 全局停止热键在 1 秒内终止新的输入动作 |
| G7 | 支持公开开源 | 具备许可证、贡献说明、风险声明和可复现构建流程 |

### 3.2 v0.1 非目标

- 不支持修改游戏内存、读取未公开接口或注入游戏进程。
- 不支持封包拦截、网络协议模拟和服务器请求伪造。
- 不提供规避反作弊、随机化以躲避检测或隐藏自动化痕迹的功能。
- 不提供账号密码托管、自动输入密码或云端代登。
- 不保证所有分辨率、语言、服务器、活动和角色阵容均可用。
- 不保证零封号、零误操作或永久兼容。
- 不自动消耗星琼、燃料、付费货币或其他高价值资源。
- 不处理抽卡、充值、商店购买、遗器分解、遗器强化等高风险功能。
- 不在 v0.1 中实现复杂地图导航和开放世界全自动寻路。

## 4. 用户与使用场景

### 4.1 目标用户

**核心用户：**
- 使用 Windows PC 玩《崩坏：星穹铁道》；
- 每天需要消耗开拓力和领取奖励；
- 能接受第三方工具带来的账号风险；
- 愿意进行首次配置和必要的异常处理。

**次级用户：**
- 开源开发者和自动化研究者；
- 希望编写、调试和分享任务流程的高级用户。

### 4.2 典型使用场景

#### 场景 A：每日一条龙

用户打开 Better Star Rail，点击“开始今日任务”。工具启动游戏并依次完成：

1. 等待用户完成登录或确认已进入游戏；
2. 检查当前界面；
3. 按配置消耗开拓力；
4. 自动完成战斗和结算；
5. 领取委托；
6. 领取每日训练；
7. 领取邮件；
8. 领取可识别的活动奖励；
9. 输出执行报告；
10. 根据设置保持游戏运行或退出游戏。

#### 场景 B：仅刷副本

用户选择副本模板、运行次数和队伍策略，点击开始。工具只执行副本循环，不领取其他奖励。

#### 场景 C：宏录制与回放

高级用户录制一段键鼠操作，编辑等待时间和循环次数，并在明确分辨率与前置界面下回放。

#### 场景 D：异常恢复

工具识别不到预期按钮、网络断开或游戏进入未知界面时，暂停输入，保存截图和日志，并提示用户接管或重试。

## 5. 产品原则

1. **确定后再点击**：识别置信度不足时不得执行高影响操作。
2. **停止优先于猜测**：未知界面连续出现后停止，而不是持续点击。
3. **每个步骤可观测**：当前任务、识别目标、重试次数和最近动作必须可见。
4. **流程可拆分**：每项任务可以单独运行、禁用和测试。
5. **默认最小权限**：非必要不要求管理员权限。
6. **本地优先**：截图、日志和设置默认只存储在本机。
7. **高风险默认关闭**：涉及不可逆资源的功能不得默认启用。
8. **版本适配明确**：每个识别资源包必须标记适用游戏版本、语言和分辨率。

## 8. 核心用户流程

### 8.1 首次启动流程

1. 展示风险声明。
2. 用户必须勾选：
   - 已理解第三方工具可能导致账号处罚；
   - 已理解项目不保证账号安全；
   - 已理解工具可能因游戏更新失效；
   - 同意高风险资源默认禁用。
3. 自动检测游戏安装位置。
4. 检测系统、分辨率、缩放比例和运行库。
5. 运行视觉校准：
   - 找到游戏窗口；
   - 检查客户端区域；
   - 识别一个基础界面元素；
   - 验证输入坐标。
6. 创建默认“一条龙”方案。
7. 进入首页。

### 8.2 一条龙执行流程

```text
开始
  ↓
环境检查
  ↓
启动/定位游戏
  ↓
等待进入主界面
  ↓
执行开拓力任务
  ↓
执行委托任务
  ↓
领取每日训练
  ↓
领取邮件
  ↓
领取活动奖励
  ↓
生成执行报告
  ↓
保持运行 / 退出游戏 / 退出工具
```

每个任务返回以下状态之一：

- Success：成功；
- Skipped：用户禁用或当前不可执行；
- Partial：部分成功；
- RecoverableError：可重试；
- FatalError：立即停止；
- UserStopped：用户终止。

### 8.3 全局停止流程

触发条件：

- 用户点击“停止”；
- 用户按下全局停止热键；
- 游戏窗口失去焦点且即将输入；
- 连续识别失败；
- 检测到付费/购买/资源补充界面；
- 运行时间超过上限；
- 游戏崩溃或进程退出。

停止后：

1. 立即停止计划中的新输入；
2. 释放按键和鼠标状态；
3. 保存最后截图；
4. 写入停止原因；
5. 弹出明确结果，不自动继续。

## 9. 详细功能需求

## 9.1 启动游戏与窗口管理

### FR-LAUNCH-001 游戏路径

- 自动检测 HoYoPlay、官方启动器和游戏可执行文件。
- 支持用户手动选择路径。
- 路径变更后重新校验。
- 不保存账号密码。

### FR-LAUNCH-002 启动模式

支持：

- 启动官方启动器；
- 启动游戏可执行文件；
- 连接已运行的游戏；
- 仅打开工具，不启动游戏。

默认优先使用官方启动器。

### FR-LAUNCH-003 窗口检测

- 通过进程和窗口标题定位游戏。
- 记录窗口客户区位置和尺寸。
- 游戏最小化时暂停视觉任务。
- 输入前确认目标窗口仍为游戏窗口。
- 不允许向非目标窗口发送自动输入。

### FR-LAUNCH-004 登录边界

- v0.1 不自动输入账号和密码。
- 检测到登录界面时提示用户手动完成。
- 用户可开启“已保存官方登录状态后自动点击进入”，但必须能够关闭。
- 遇到验证码、人机验证或安全验证时立即停止并交由用户处理。

### 验收标准

- 在已配置路径下，启动成功率不低于 95%。
- 游戏窗口切换后，不得误向其他应用输入。
- 无法定位游戏时，错误信息必须可理解并提供修复入口。

## 9.2 视觉识别底座

### FR-VISION-001 屏幕采集

- 支持游戏窗口客户区截图。
- 支持指定 ROI 截图。
- 截图坐标必须基于客户区，不基于屏幕绝对坐标。
- 支持保存失败帧。
- 默认不长期保存连续截图。

### FR-VISION-002 模板匹配

每个模板包含：

- 模板 ID；
- 名称；
- 适用游戏版本；
- 语言；
- 基准分辨率；
- ROI；
- 置信度阈值；
- 缩放范围；
- 失败策略。

### FR-VISION-003 OCR

- v0.1 首先支持简体中文。
- OCR 可限定识别区域。
- 输出文字、边界框和置信度。
- 支持文本标准化，如去空格、全半角和常见误识别修正。
- 目标文字必须支持精确匹配和关键词匹配。
- OCR 点击前需同时满足区域和置信度条件。

### FR-VISION-004 界面状态识别

基础状态至少包括：

- 启动器；
- 登录页；
- 主界面；
- 手机菜单；
- 地图；
- 开拓力副本入口；
- 队伍选择；
- 战斗中；
- 战斗胜利；
- 战斗失败；
- 奖励结算；
- 委托；
- 每日训练；
- 邮件；
- 活动；
- 资源不足；
- 网络错误；
- 未知界面。

### FR-VISION-005 调试视图

高级设置可开启：

- ROI 边框；
- 匹配模板名称；
- 识别置信度；
- OCR 文本框；
- 最近识别耗时；
- 当前状态机状态。

### 验收标准

- 标准 1920×1080、100% 缩放、简体中文环境为 v0.1 基准环境。
- 基准环境中核心按钮识别准确率不低于 95%。
- 未知界面不得自动执行连续盲点操作。
- OCR 置信度不足时必须停止或降级为人工确认。

## 9.3 键鼠输入底座

### FR-INPUT-001 输入动作

支持：

- 鼠标移动；
- 左键单击；
- 双击；
- 鼠标按下/释放；
- 键盘按下/释放；
- 组合键；
- 固定等待；
- 条件等待；
- 输入动作序列。

### FR-INPUT-002 坐标转换

- 所有点击坐标基于窗口客户区比例坐标或识别框中心。
- 支持窗口移动后自动换算。
- 不以写死的屏幕绝对坐标作为默认实现。

### FR-INPUT-003 安全延迟

- 每次关键点击后必须等待界面变化。
- 禁止用极短无条件循环持续点击。
- 支持操作超时和最大重试次数。
- 运行速度可调，但不得突破最小安全间隔。

### FR-INPUT-004 停止热键

- 默认全局停止热键：`Ctrl + Shift + F12`。
- 用户可修改。
- 按下后 1 秒内停止新的输入。
- 停止时强制释放所有可能处于按下状态的按键。

## 9.4 任务调度器

### FR-SCHED-001 任务编排

每个任务配置：

- 启用状态；
- 执行顺序；
- 前置条件；
- 最大运行时间；
- 最大重试次数；
- 失败后继续或停止；
- 完成后动作；
- 是否需要人工确认。

### FR-SCHED-002 断点与恢复

- 每完成一个子任务写入检查点。
- 应用异常退出后，下次启动显示未完成任务。
- 默认由用户选择“重新开始”或“从安全检查点继续”。
- 不允许从战斗中、购买界面或未知界面直接恢复。

### FR-SCHED-003 运行模式

- 完整一条龙；
- 仅运行选中任务；
- 调试单步；
- 从当前界面开始；
- 定时运行。

### FR-SCHED-004 防重复

- 每日任务记录当日完成状态。
- 已完成任务默认跳过。
- 用户可强制重新运行。
- 防重复逻辑只作为提示，不能假设游戏状态绝对正确。

## 9.5 自动消耗开拓力

### FR-POWER-001 副本配置

用户可配置：

- 副本类型；
- 目标副本；
- 单次消耗；
- 计划次数；
- 最大开拓力消耗；
- 是否使用支援；
- 是否允许重复挑战；
- 体力不足后的行为。

v0.1 优先支持用户预先选择并验证过的固定副本流程。

### FR-POWER-002 开拓力保护

默认规则：

- 不使用燃料；
- 不使用星琼补充开拓力；
- 不购买额外次数；
- 不使用任何付费或稀有补充资源；
- 检测到补充界面立即停止并记录。

### FR-POWER-003 次数策略

支持：

- 固定次数；
- 尽可能消耗当前自然恢复开拓力；
- 消耗到用户设定下限；
- 单次测试。

### FR-POWER-004 副本进入

流程：

1. 检查主界面；
2. 打开指定入口；
3. 定位目标副本；
4. 确认消耗数值；
5. 选择挑战次数；
6. 进入队伍界面；
7. 确认队伍可用；
8. 开始挑战。

无法确认目标副本名称或资源消耗时不得开始。

### FR-POWER-005 结束条件

- 达到次数；
- 开拓力不足；
- 连续战斗失败；
- 队伍不可用；
- 网络错误；
- 用户中止；
- 超时；
- 检测到高风险资源弹窗。

### 验收标准

- 不得自动使用燃料或星琼。
- 计划次数和实际次数必须记录。
- 进入错误副本时必须能够在消耗资源前停止。
- 资源不足时不得陷入循环点击。

## 9.6 自动战斗与副本循环

### FR-BATTLE-001 战斗开始

- 识别队伍界面和开始挑战按钮。
- 开始前检查队伍是否存在明显不可用状态。
- v0.1 不自动更换复杂配队，仅允许用户选择预设队伍槽位。
- 支援角色选择可作为 P1 功能。

### FR-BATTLE-002 自动战斗

- 进入战斗后识别自动战斗状态。
- 若未开启，可按用户设置开启。
- 不在剧情、教程或不可自动战斗场景中反复尝试。

### FR-BATTLE-003 战斗状态

必须识别：

- 加载；
- 战斗中；
- 胜利；
- 失败；
- 结算；
- 再次挑战；
- 退出；
- 网络重连。

### FR-BATTLE-004 战斗失败

默认策略：

1. 第一次失败：保存截图并重试一次；
2. 第二次失败：停止当前副本；
3. 是否继续执行其他奖励任务由用户设置决定。

禁止无限重试。

### FR-BATTLE-005 结算循环

- 识别奖励结算页面；
- 记录完成次数；
- 若满足继续条件，点击“再次挑战”；
- 否则退出副本；
- 退出后验证回到安全界面。

### 验收标准

- 战斗结束后 30 秒内进入下一状态或报错。
- 连续未知状态超过设定阈值必须停止。
- 失败重试不得超过用户设置上限。

## 10. 异常处理规范

| 异常 | 默认处理 |
|---|---|
| 游戏未启动 | 尝试启动一次，失败后提示 |
| 游戏更新 | 停止并提示用户完成更新 |
| 登录失效 | 暂停，要求用户登录 |
| 验证码/安全验证 | 立即停止人工处理 |
| 网络断开 | 尝试游戏内重连一次 |
| 游戏窗口最小化 | 暂停输入 |
| 游戏窗口失焦 | 输入前重新确认，不自动抢焦点超过 1 次 |
| 分辨率变化 | 停止并要求重新校准 |
| OCR 置信度低 | 重试截图，仍失败则停止模块 |
| 模板连续失败 | 保存截图并停止模块 |
| 资源不足 | 退出副本流程，不补充付费资源 |
| 战斗失败 | 有限重试，超过上限停止 |
| 背包满 | 停止相关领取，不自动分解物品 |
| 未知弹窗 | 不点击，截图并停止 |
| 工具崩溃 | 保存崩溃日志，下次显示恢复提示 |

## 11.1 产品级红线

- 游戏进程注入；
- DLL 注入；
- 内存扫描、内存读写或修改；
- 驱动级操作；
- 封包抓取、篡改或伪造；
- 反作弊禁用、卸载、绕过或干扰；
- 进程隐藏、窗口隐藏以规避检测；

- 账号凭据收集、上传或共享；

- 远程执行任意代码；

## 12. 非功能需求

### 12.1 兼容性

v0.1 正式支持基线：

- Windows 10/11 x64；
- .NET 8；
- 简体中文游戏界面；
- 1920×1080；
- Windows 显示缩放 100%；
- 游戏窗口模式或无边框窗口；
- 单显示器为主要测试环境。

其他环境可标记为实验性：

- 2560×1440；
- 125% / 150% 缩放；
- 多显示器；
- 国际服；
- 其他语言。

### 12.2 性能

- 空闲内存占用目标：低于 300 MB；
- 自动化运行内存占用目标：低于 800 MB；
- 平均 CPU 占用目标：低于 15%，识别瞬时峰值除外；
- 常规识别响应：1 秒内；
- 全局停止响应：1 秒内；
- 工具不得导致游戏帧率长期明显下降。

### 12.3 稳定性

- 单次一条龙最长运行默认 90 分钟；
- 单任务必须有超时；
- 所有循环必须有上限或明确退出条件；
- 未处理异常必须写入崩溃日志；
- 关键配置写入采用原子保存，避免断电损坏。

### 12.4 可维护性

- 任务逻辑、识别资源和 UI 解耦；
- 每个任务拥有独立状态机；
- 识别模板通过资源包版本管理；
- 错误码统一；
- 核心流程具备单元测试和可回放截图测试；
- 不依赖只能由单一开发者维护的本地路径或私有服务。

### 12.5 可访问性与易用性

- 关键按钮具备文字，不只使用图标；
- 风险和错误使用清楚语言；
- 首页不展示过量开发参数；
- 高级设置与普通设置分离；
- 支持中文界面；
- 日志可复制和导出。

- 有开拓力 / 无开拓力；
- 战斗胜利 / 战斗失败；
- 有奖励 / 无奖励；
- 网络正常 / 短暂断网；
- 游戏失焦 / 最小化；
- 未知弹窗；
- 登录过期。

### 15.2 测试类型

- 单元测试：坐标换算、配置、状态转换；
- 截图回放测试：使用固定截图验证识别；
- 集成测试：任务调度和输入模拟；
- 端到端测试：完整一条龙；
- 异常注入测试：断网、失焦、资源不足；
- 安全测试：高风险页面拦截；
- 回归测试：游戏版本更新后核心流程。

### 15.3 发布前人工检查

- 从全新 Windows 用户环境安装；
- 不预装开发环境时可正常启动或给出依赖提示；
- 首次风险声明可见；
- 默认高风险功能关闭；
- 停止热键有效；
- 日志路径可访问；
- 卸载后不残留计划任务；
- Release 包与源码构建结果一致；
- 病毒扫描和依赖审计无已知高危问题。

## 17. 技术实现约束

本节不是完整技术设计，但作为产品约束写入 PRD。

### 17.1 推荐技术栈

- C# / .NET 8；
- WPF 桌面 UI；
- MVVM；
- OpenCV 兼容库用于模板匹配；
- 可替换 OCR 引擎接口；
- Windows 输入模拟接口；
- 结构化日志；
- JSON 配置和资源清单。

### 17.2 模块边界

建议模块：

- `AppShell`
- `GameProcessService`
- `WindowCaptureService`
- `VisionService`
- `OcrService`
- `InputService`
- `TaskScheduler`
- `SafetyGuard`
- `TaskModules`
- `MacroEngine`
- `ResourcePackManager`
- `Logging`
- `Settings`
- `UpdateCompatibility`

### 17.3 强制规则

- Task 模块不能直接调用系统输入 API，必须经过 `InputService`。
- 所有输入必须经过 `SafetyGuard`。
- 所有视觉识别必须返回置信度和证据区域。
- 所有循环必须支持取消令牌。
- 所有任务必须有超时。
- 任何高风险界面识别结果优先级高于普通任务。
- 日志不得写入密码、Token 或支付信息。

## 19. 风险清单

| 风险 | 影响 | 概率 | 缓解措施 |
|---|---|---:|---|
| 账号被处罚 | 极高 | 中高 | 明示风险、不承诺安全、不提供规避检测 |
| 游戏更新导致识别失效 | 高 | 高 | 资源包版本化、失败即停、回归测试 |
| OCR 误识别导致误点 | 高 | 中 | ROI、置信度、多条件确认、高风险拦截 |
| 分辨率和缩放差异 | 高 | 高 | v0.1 限定基准环境、校准、比例坐标 |
| 未知弹窗导致卡死 | 中高 | 高 | 未知状态超时、截图、停止 |
| 无限循环 | 高 | 中 | 所有循环有次数和总超时 |
| 用户无人值守时异常 | 高 | 中 | 运行上限、停止条件、完成报告 |
| 第三方依赖许可证冲突 | 高 | 中 | 依赖清单、许可证审查 |
| 游戏素材知识产权风险 | 中高 | 中 | 最小裁剪、避免大规模素材分发 |
| 被误认为官方工具 | 中 | 中 | 非官方标识、独立品牌和图标 |
| 恶意宏文件 | 高 | 中 | 受限格式、无系统命令、导入审计 |
| 管理员权限扩大风险 | 高 | 中 | 默认普通权限、按需提示 |
| 用户日志泄露 UID | 中 | 中 | 脱敏提醒、本地保存 |

## 20. 验收清单

### 产品验收

- [ ] 首次启动展示风险声明并记录本地确认。
- [ ] 能检测或配置游戏路径。
- [ ] 能启动游戏或连接已运行游戏。
- [ ] 能识别主界面。
- [ ] 能运行一个完整的开拓力副本循环。
- [ ] 能识别战斗胜利、失败和结算。
- [ ] 能领取委托并重新派遣。
- [ ] 能领取每日训练。
- [ ] 能领取邮件附件且不删除邮件。
- [ ] 能生成执行报告。
- [ ] 能通过全局热键停止。
- [ ] 失焦时不向其他窗口输入。
- [ ] 检测到开拓力补充/购买界面时停止。
- [ ] 所有任务有超时和重试上限。
- [ ] 未知界面能够保存截图并停止。
- [ ] 默认不保存账号凭据。
- [ ] 默认不上传截图或日志。
- [ ] 开源许可证与第三方许可证完整。
- [ ] README 明确支持环境和已知限制。
- [ ] Release 包可以从公开源码复现构建。

### 质量否决项

出现以下任一情况不得发布：

- 自动使用星琼、燃料或付费货币；
- 向非游戏窗口输入；
- 无法通过热键停止；
- 存在无限循环；
- 登录凭据被记录；
- 存在反作弊绕过或规避检测模块；
- 未披露账号处罚风险；
- 使用来源不明或许可证不兼容的代码和素材；
- 安装包与公开源码不一致；
- 已知高危依赖未处理。

## 22. PRD 评审结论建议

建议批准 v0.1 进入下一阶段，但将其定义为**公开预览版，而不是稳定版**。

批准条件：

- 接受“无法保证账号安全”的产品事实；
- 严格执行不注入、不读写内存、不改封包、不绕过反作弊的边界；
- 先完成固定环境、固定副本和固定语言的可靠闭环；
- 不在第一版追求全部活动和全部分辨率；
- 将安全停止、日志和异常恢复视为 P0，而不是上线后的补充功能。

下一份正式文档应为：

> 《Better Star Rail v0.1 技术设计文档（TDD）与模块拆分》

其中需要确定状态机、项目目录、接口、依赖、任务模块协议、截图识别测试框架和第一阶段开发任务。
